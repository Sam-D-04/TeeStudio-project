Để hệ thống thực sự hoàn hảo khi ra hội đồng bảo vệ, nhóm nên cân nhắc tinh chỉnh thêm 4 điểm sau:
A. Đổi LONGTEXT thành JSON
●	Hiện tại: Bảng CustomDesign đang dùng ``canvasData LONGTEXT.
●	Góp ý: Vì dự án dùng MySQL 8.0, hãy đổi thành ``canvasData JSON. MySQL tối ưu truy vấn JSON cực tốt và Prisma ORM cũng có hàm hỗ trợ mapping trực tiếp kiểu JSON, giúp Backend không phải dùng lệnh JSON.parse() hay JSON.stringify() thủ công nữa.
B. Thiếu thông tin Đơn vị Vận chuyển (Shipping)
●	Hiện tại: Bảng CustomerOrder có địa chỉ (addressId) nhưng thiếu thông tin giao hàng.
●	Góp ý: Quản trị viên cần nhập mã vận đơn để khách theo dõi. Nên bổ sung vào bảng CustomerOrder các cột: shippingFee (DECIMAL - Phí ship), shippingCarrier (VARCHAR - Đơn vị vận chuyển VD: GHTK, GHN), trackingCode (VARCHAR - Mã vận đơn).
C. Mở rộng cấu hình phụ phí (Print Method)
●	Hiện tại: Nhóm đang dùng PrintPosition (Vị trí in) để tính thêm tiền.
●	Góp ý: Theo UI Admin, chúng ta có tính thêm tiền theo Công nghệ in (In PET, In Lụa). Bạn nên thêm bảng PrintMethod (id, name, extraCost) hoặc đổi tên bảng PrintPosition thành PrintModifier (Tùy chọn in ấn) để bao hàm cả vị trí, công nghệ và kích thước logo.
D. Bù đắp lỗ hổng Quản lý Nhập kho (Supplier)
●	Hiện tại: Bảng InventoryTransaction có lưu lịch sử xuất nhập, nhưng nhóm đã xóa mất bảng NhaCungCap (Supplier).
●	Góp ý: Nếu xưởng nhập phôi áo từ các xưởng may khác, Kế toán sẽ cần biết mình đang nợ tiền ai. Nên bổ sung lại bảng Supplier (id, name, phone) và thêm cột supplierId (có thể NULL) vào bảng InventoryTransaction đối với các giao dịch có transactionType = 'IMPORT'.



3. Các lỗi quan trọng trong file mới cần sửa ngay
Lỗi 1: Cột NOT NULL nhưng khóa ngoại lại ON DELETE SET NULL
Trong file mới có nhiều cột khai báo NOT NULL, nhưng khóa ngoại lại ghi ON DELETE SET NULL. Đây là mâu thuẫn logic. Khi bản ghi cha bị xóa, database muốn set cột con về NULL, nhưng cột đó lại không cho phép NULL.
Các cột đang bị lỗi:
ProductImage.variantId INTEGER NOT NULL
CartItem.designId INTEGER NOT NULL
CustomerOrder.promotionId INTEGER NOT NULL
OrderItem.designId INTEGER NOT NULL
InventoryTransaction.orderId INTEGER NOT NULL
Nhưng bên dưới lại có:
ON DELETE SET NULL
Nên sửa thành:
ProductImage.variantId INTEGER NULL
CartItem.designId INTEGER NULL
CustomerOrder.promotionId INTEGER NULL
OrderItem.designId INTEGER NULL
InventoryTransaction.orderId INTEGER NULL
Lý do:
●	Không phải ảnh sản phẩm nào cũng gắn với một biến thể cụ thể.
●	Không phải khách nào cũng dùng mã khuyến mãi.
●	Không phải đơn nào cũng có áo custom nếu hệ thống có bán áo mẫu.
●	Không phải giao dịch kho nào cũng liên quan đến đơn hàng, ví dụ nhập kho thủ công.
________________________________________
Lỗi 2: Thiếu UNIQUE cho các trường quan trọng
File mới có nhiều cột cần không được trùng nhưng chưa khai báo UNIQUE.
Nên thêm unique cho:
Account.email
Product.slug
ProductVariant.sku
Promotion.code
CustomerOrder.orderCode
Nếu không có unique, hệ thống có thể bị:
●	2 tài khoản cùng email.
●	2 sản phẩm cùng slug.
●	2 biến thể cùng SKU.
●	2 mã đơn hàng giống nhau.
●	2 mã giảm giá giống nhau.
Gợi ý thêm:
ALTER TABLE Account ADD UNIQUE (`email`);
ALTER TABLE Product ADD UNIQUE (`slug`);
ALTER TABLE ProductVariant ADD UNIQUE (`sku`);
ALTER TABLE Promotion ADD UNIQUE (`code`);
ALTER TABLE CustomerOrder ADD UNIQUE (`orderCode`);
________________________________________
Lỗi 3: updatedAt chưa tự cập nhật
Hiện tại nhiều bảng có:
updatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
Nhưng như vậy chỉ tự set lúc tạo, không tự đổi khi update.
Nên sửa thành:
updatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
Áp dụng cho:
Account
CustomDesign
Cart
CustomerOrder
________________________________________
Lỗi 4: CartItem.designId và OrderItem.designId đang bắt buộc
Hiện tại:
designId INTEGER NOT NULL
Điều này chỉ đúng nếu website chỉ bán áo tự thiết kế, không bán áo mẫu/sản phẩm có sẵn.
Nếu web có cả 2 kiểu:
1.	Áo có sẵn.
2.	Áo tự thiết kế.
Thì designId phải là NULL.
Nên sửa:
designId INTEGER NULL
Và hiểu logic như sau:
designId = NULL     => mua áo thường
designId != NULL    => mua áo custom
________________________________________
Lỗi 5: Payment.transactionId không nên bắt buộc
Hiện tại:
transactionId VARCHAR(255) NOT NULL
Nhưng nếu khách chọn COD, chuyển khoản chờ xác nhận, hoặc payment đang pending thì có thể chưa có mã giao dịch.
Nên sửa:
transactionId VARCHAR(255) NULL
Nên thêm thêm:
paidAt DATETIME NULL
gatewayResponse TEXT NULL

4. Các phần còn thiếu so với web bán áo tự thiết kế thực tế
Thiếu bảng/lưu trạng thái sản xuất áo
Web áo custom không chỉ có đặt hàng, mà còn có quy trình sau đơn hàng:
Khách đặt hàng -> duyệt thiết kế -> in áo -> đóng gói -> giao hàng -> hoàn tất
Hiện tại CustomerOrder.status có thể lưu trạng thái đơn, nhưng chưa đủ chi tiết cho sản xuất.
Nên thêm bảng kiểu:
OrderProduction
- id
- orderItemId
- designId
- status
- note
- approvedAt
- printedAt
- packedAt
- createdAt
Hoặc tối thiểu thêm trạng thái rõ trong OrderItem, ví dụ:
productionStatus VARCHAR(30)
Các trạng thái nên có:
WAITING_DESIGN_APPROVAL
APPROVED
PRINTING
PRINTED
PACKED
DONE
REJECTED
________________________________________
Thiếu thông tin giao hàng
CustomerOrder hiện có addressId, nhưng chưa có phí ship, đơn vị vận chuyển, mã vận đơn.
Nên thêm:
shippingFee DECIMAL(15,2) NOT NULL DEFAULT 0
shippingMethod VARCHAR(50) NULL
trackingCode VARCHAR(100) NULL
shippedAt DATETIME NULL
deliveredAt DATETIME NULL
cancelReason TEXT NULL
Vì web bán áo online gần như chắc chắn cần quản lý giao hàng.
________________________________________
CustomDesign nên liên kết rõ với màu/variant áo
Hiện tại CustomDesign có:
productId
baseColor
Còn OrderItem lại có:
variantId
designId
Có thể xảy ra lỗi: khách thiết kế áo màu trắng nhưng khi đặt hàng lại chọn variant màu đen.
Nên cân nhắc sửa CustomDesign thành:
variantId INTEGER NULL
Hoặc giữ productId nhưng phải validate trong backend rằng:
CustomDesign.productId phải khớp với ProductVariant.productId
CustomDesign.baseColor phải khớp với ProductVariant.color
________________________________________
PrintPosition còn hơi đơn giản
Hiện tại có:
name
extraCost
Với web thiết kế áo, nên có thêm:
code
maxWidth
maxHeight
printAreaX
printAreaY
printAreaWidth
printAreaHeight
Ví dụ vị trí in:
FRONT
BACK
LEFT_CHEST
RIGHT_SLEEVE
Như vậy frontend canvas sẽ biết vùng nào được phép đặt hình/text.
